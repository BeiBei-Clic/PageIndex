我们首先来看背景知识。

----------------------------------------------------------------
第 3 页 | 程序执行路径与三种基本结构
----------------------------------------------------------------

任何一个程序的执行路径，都可以由三种基本控制结构组合而成：顺序结构，语句从上到下依次执行；选择结构，也就是 if-else 条件分支，根据条件决定走哪条路；以及循环结构，包括 for 循环和 while 循环，重复执行某段代码。

代码覆盖率的核心思想就是：程序中所有可能的分支路径构成了"分母"，实际测试执行到的路径构成了"分子"，两者的比值就是覆盖率。所以公式可以写成：覆盖率 = 已执行分支数 / 总分支数 × 100%。

那么路径分析和软件测试到底是什么关系呢？我们来看下一页。

----------------------------------------------------------------
第 4 页 | 路径分析与软件测试的关系
----------------------------------------------------------------

路径分析在软件测试中扮演着关键角色。一个完整的测试流程通常包含五个步骤：首先理解程序逻辑，然后识别所有可能的路径，接着设计测试用例覆盖这些路径，再执行测试并收集数据，最后评估覆盖率是否达标。

我们这个项目关注的是后三个环节——即自动识别路径、追踪执行、计算覆盖率。目标是让这个过程自动化，不需要人工逐行检查。

那么整个工具的流程是怎样的呢？

----------------------------------------------------------------
第 5 页 | 整体流程概览
----------------------------------------------------------------

我们的工具分为三个阶段。第一阶段是静态分析，通过解析源代码的抽象语法树（AST），识别出所有分支结构，包括 if、for、while、try-except 等。第二阶段是动态追踪，利用 Python 的 sys.settrace 钩子，在程序运行时记录实际执行到了哪些行。第三阶段是覆盖率计算，将静态分析得到的"应该有的分支"和动态追踪得到的"实际走过的分支"进行对比，生成覆盖率报告。

接下来我们详细介绍每个阶段的技术细节。

----------------------------------------------------------------
第 6 页 | 静态分析（1/2）：AST
----------------------------------------------------------------

静态分析的第一步是构建抽象语法树（AST）。什么是 AST 呢？当我们给 Python 解释器一段源代码时，解释器会先把源码解析成一棵树形结构，这棵树就是 AST。树上的每个节点对应源码中的一个语法元素——比如一个 if 语句是一个节点，一个 for 循环也是一个节点。

幻灯片上展示了一个简单的示例。左边是一段包含 if-else 的代码，右边是对应的 AST 片段。可以看到 if 语句被解析成了一个 If 节点，它包含了 test（条件）、body（then 分支）和 orelse（else 分支）三个子节点。通过遍历这棵树，我们就能识别出所有的分支点。

具体在代码中是怎么做的呢？

----------------------------------------------------------------
第 7 页 | 静态分析（2/2）：cfg_builder.py
----------------------------------------------------------------

这是我们项目中 cfg_builder.py 的核心代码。CFGBuilder 类的 build_from_file 方法接收一个文件路径，先读取源码，然后用 ast.parse 把它解析成 AST。接下来遍历 AST 中的每个节点，在 _extract_branch 方法中判断节点类型：如果是 ast.If，就记录为一个 IF 类型的分支；如果是 ast.For，记录为 FOR 类型；以此类推，还有 While 和 ExceptHandler。

每个识别到的分支会被封装成一个 CodeBranch 对象，保存文件路径、行号、分支类型和结束行号。这样，静态分析的输出就是一个完整的分支列表。

在讲动态追踪之前，需要先了解一些 Python 运行机制的背景知识。

----------------------------------------------------------------
第 8 页 | 前置知识（1/2）：源码→字节码
----------------------------------------------------------------

Python 程序的执行分为两步。第一步，源代码被编译成字节码（bytecode）。字节码是一种中间表示，每条指令对应一个简单的操作，比如加载变量、比较大小、跳转等。我们可以用 Python 内置的 dis 模块来查看一段代码对应的字节码。

这个编译过程就是从 .py 文件到 .pyc 文件的过程。了解字节码的概念有助于理解下一页讲的虚拟机执行机制。

----------------------------------------------------------------
第 9 页 | 前置知识（2/2）：虚拟机与栈帧
----------------------------------------------------------------

第二步，Python 虚拟机（也就是 PVM）逐条执行字节码。执行的核心数据结构是栈帧（frame）。每个函数调用都会创建一个新的栈帧，栈帧里保存了局部变量、字节码指令指针、操作数栈等信息。

多个栈帧组成了调用栈——当函数 A 调用函数 B 时，B 的栈帧被压到 A 的栈帧上面；当 B 返回时，它的栈帧被弹出。Python 提供的 sys.settrace 正是基于栈帧来工作的，这也是我们动态追踪的基础。

好，接下来我们看动态追踪具体是怎么实现的。

----------------------------------------------------------------
第 10 页 | 动态追踪（1/2）：sys.settrace()
----------------------------------------------------------------

sys.settrace() 是 Python 提供的一个调试钩子。当我们注册一个回调函数后，Python 虚拟机在执行每条字节码之前都会调用这个回调，并传入三个参数：frame（当前栈帧）、event（事件类型）和 arg（事件参数）。

主要的事件类型有三种："call" 表示进入了一个函数；"line" 表示执行到了新的一行；"return" 表示函数即将返回。通过监听这些事件，我们就能精确地知道程序执行了哪些行、调用了哪些函数。

具体代码实现如下。

----------------------------------------------------------------
第 11 页 | 动态追踪（2/2）：tracer.py
----------------------------------------------------------------

这是 tracer.py 的核心实现。PathTracer 类使用了上下文管理器模式，进入 with 语句块时通过 sys.settrace 注册回调，退出时恢复原来的追踪函数。

回调函数 _trace_callback 里，首先检查当前文件是否是我们要追踪的目标文件，避免记录不相关的库函数。然后根据事件类型分别处理："line" 事件时，把当前行号加入已执行行集合；"call" 事件时，提取函数名、参数和调用深度，构建 FunctionCall 对象，维护一个调用栈来记录嵌套关系；"return" 事件时，从调用栈弹出并记录返回值。

这样，动态追踪结束后，我们就有了完整的执行路径数据。

有了静态分析结果和动态追踪数据，接下来就是计算覆盖率了。

----------------------------------------------------------------
第 12 页 | 覆盖率计算与报告生成
----------------------------------------------------------------

覆盖率计算的核心逻辑在 reporter.py 中。它的思路很直观：先用 CFGBuilder 做静态分析，得到所有的分支列表；然后从 tracer 获取实际执行到的行号集合；接着遍历每个分支，检查该分支的起始行号是否在已执行行集合中，如果是就标记为"已执行"。

最后统计总数：已执行分支数除以总分支数，就得到了覆盖率百分比。报告支持三种格式输出——纯文本、JSON 和 HTML。HTML 格式还会高亮源码，绿色表示已执行，红色表示未执行。

了解了各模块的功能后，我们来看项目的整体结构。

----------------------------------------------------------------
第 13 页 | 项目模块结构
----------------------------------------------------------------

项目由六个核心文件组成。models.py 定义了所有的数据模型，也就是分支、函数调用、执行路径和覆盖率报告这些数据结构。cfg_builder.py 负责静态分析，从源码中提取分支。tracer.py 负责动态追踪，记录运行时的执行路径。reporter.py 是报告生成器，对比静态和动态数据并输出报告。cli.py 提供命令行接口。demo.py 是演示脚本，包含各种测试用例。

它们之间的协作关系是：cfg_builder 和 tracer 分别独立工作，一个看源码，一个看运行时；最后由 reporter 把两者的结果汇总。

下面我们来看数据模型的详细定义。

----------------------------------------------------------------
第 14 页 | 数据模型（1/2）
----------------------------------------------------------------

首先是 BranchType 枚举，定义了我们支持的所有分支类型：SEQUENTIAL、IF、ELSE、ELIF、FOR、WHILE、TRY 和 EXCEPT，共 8 种。

然后是 CodeBranch 数据类，表示一个分支点。它包含五个字段：file_path 是文件路径，line_no 是分支起始行号，branch_type 是分支类型，end_line 是结束行号，is_executed 标记该分支在运行时是否被执行到。

接下来看更复杂的数据模型。

----------------------------------------------------------------
第 15 页 | 数据模型（2/2）
----------------------------------------------------------------

FunctionCall 记录一次函数调用，包含函数名、文件路径、行号、调用深度、参数字典、返回值，以及一个 children 列表用来表示嵌套调用关系——比如 A 调用了 B，B 就是 A 的 child。

ExecutionPath 记录一个文件的完整执行路径，包含已执行行号集合、已执行分支字典和函数调用列表。

CoverageReport 是最终汇总的覆盖率报告，包含总分支数、已执行分支数、各类型分支计数、覆盖率百分比，以及每个文件的详细报告。

第 16 页 | 被追踪的示例代码（1/2）

demo.py 中定义了多个测试函数。第一个是 demo_conditionals，它接受一个整数参数，通过一系列 if-elif-else 条件判断，将数值分类为 negative、zero、small、medium 或 large。这个函数专门用来展示条件分支的覆盖情况——如果只传入一个正数，那么 negative 和 zero 的分支就走不到。

demo_for_loop 是一个简单的求和函数，遍历列表累加。demo_while_loop 用 while 循环生成一个等差数列。

demo_nested_structures 展示了嵌套结构——for 循环里面嵌套了 while 循环，而且外层有 if-else 判断。这种嵌套场景对追踪工具是一个很好的测试。

第 20 页 | 端到端流程（3/3）
----------------------------------------------------------------

第三步，覆盖率计算。CoverageReporter 接收静态分支列表和动态执行数据，逐一比对。如果某个分支的起始行号出现在已执行行集合中，就标记为已执行。最终汇总统计：总共有多少个分支、执行了多少个、覆盖率是多少。同时支持输出文本报告、JSON 和 HTML 格式。

第 25 页 | PageIndex 项目简介
----------------------------------------------------------------

PageIndex 是一个无向量文档索引系统。它的核心思路是不使用传统的向量嵌入，而是将文档解析成树形结构索引，然后利用大语言模型来做文档检索和问答。

项目的主要模块包括：page_index.py 负责文档索引构建，search.py 是核心搜索逻辑，agent_pageindex.py 是基于 LangChain 的对话式入口。我们选取 PageIndex 作为被测对象，因为它的代码规模适中，而且包含各种控制结构。

先来看它的核心搜索流程。

----------------------------------------------------------------
第 26 页 | 核心搜索流程（search.py）
----------------------------------------------------------------

search.py 中的 run_tree_search 函数是整个系统的入口。它的前半部分流程是：首先从数据库加载文档目录列表，每个条目包含文档 ID、名称和描述。然后将这些信息交给 LLM，让 LLM 判断哪些文档可能与用户查询相关，选出 top-k 个文档。这个步骤相当于一个文档初筛，避免对无关文档做无谓的搜索。

----------------------------------------------------------------
第 27 页 | 核心搜索流程续（search.py）
----------------------------------------------------------------

run_tree_search 的后半部分：对于 LLM 选出的文档，从数据库加载它们的完整树形索引，然后并发地调用 LLM 对每棵树做更精细的搜索，找出与查询相关的节点。搜索结果汇总后，拼装成上下文，再交给 LLM 生成最终答案。整个流程是"粗筛→精搜→生成"的三段式架构。
